
# Bug 2: 子 agent 触发的 circular import（runtime ⇄ tools）

## 触发点

程序在 **import 阶段就崩**，还没进入 agent loop。`tools/multi_agents.py` 在模块顶层写了：

```python
from my_claude_code.runtime import create_runtime
```

启动时报：

```text
ImportError: cannot import name 'create_runtime' from partially initialized
module 'my_claude_code.runtime' (most likely due to a circular import)
```

## 根源

`runtime` 和 `tools` 互相依赖，在模块加载期形成闭环：

```text
runtime.py
  └─ from my_claude_code.tools import *          # 顶层，加载整个 tools 包
       └─ tools/__init__ → registry
            └─ registry 收集各 tool 的 SPECS，import 了 multi_agents
                 └─ multi_agents.py 顶层: from my_claude_code.runtime import create_runtime
                      └─ 回到 runtime —— 但它还停在上面那行没返回，
                         create_runtime（定义在文件末尾）此刻还不存在 → ImportError
```

关键点：`create_runtime` 定义在 `runtime.py` 末尾，而上面的 import 链在 runtime「只执行到顶部的 import」时就回头来取它，此时模块处于 *partially initialized* 状态，名字还没绑定。

## 修改前

```python
# tools/multi_agents.py —— 顶层 import，模块一加载就执行 → 撞环
from my_claude_code.runtime import create_runtime

def run_subagent(ctx, prompt, max_turn=50):
    subagent = create_runtime("subagent", ...)
    ...
```

## 修改后

```python
# tools/multi_agents.py —— 顶层不再 import runtime
def run_subagent(ctx, prompt, max_turn=50):
    from my_claude_code.runtime import create_runtime   # 延迟到「调用时」才 import
    subagent = create_runtime("subagent", ...)
    ...
```

等到真正调用 `run_subagent` 时，runtime 和 tools 都早已完成加载，环自然解开。

## 结论

- **直接修法是 lazy import**：把「被依赖方」的 import 从模块顶层挪进函数体，避开加载期的闭环。这是止血，且对低频调用（spawn 子 agent）几乎没有性能代价。
- **同一个矛盾的另一处痕迹**：`contracts.py` 里 `ToolContext.runtime: Any  # Avoid circular import` —— tools 层想引用 `Runtime` 类型，但顶层 import 它又会成环，于是退而用 `Any`（代价：丢了静态类型）。
- **更根本的解法是反转依赖方向**：让 `tools` 不认识 `runtime`。例如把 `create_runtime` 由上层注入、或把子 agent 的创建挪出 tools 层，让 tools 只依赖 `contracts` 这种纯数据契约。lazy import 越多，越说明分层该重画了。
- 教训：和 Bug 1 一样，根子在「先 copy 跑通、分层没想清楚」。circular import 不是 Python 的毛病，是依赖图有环的信号。
